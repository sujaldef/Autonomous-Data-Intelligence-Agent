ADIA Backend Implementation Guide and Frontend Integration Audit

Status of the current repository:
- The frontend is present and renders multiple project views, but it is currently UI-only.
- The backend directory contains only an empty server.py and an empty utils/db.py, plus empty controller, model, and routes folders.
- The backend therefore needs a complete implementation plan, not just an explanation of existing code.
- This document describes the backend that must be built so that every visible frontend feature has a concrete API and data source.
- It also identifies the missing endpoints, support services, data models, and security primitives required for production use.

Primary backend design goals:
- Provide deterministic, auditable responses to user queries.
- Keep the LLM as a planner, not as an executor of direct side effects.
- Support authentication, project dashboards, query execution, history, analytics, and notification management.
- Preserve a strict boundary between read operations, analysis operations, and mutation operations.
- Make the backend implementation-ready so each file has explicit responsibilities, data flow, and interfaces.

Global backend assumptions:
- FastAPI is the web framework.
- Pydantic models are used for validation and serialization.
- SQLAlchemy is used for database access.
- SQLite can be the first storage engine, but the file structure must allow an easy migration to PostgreSQL.
- LangChain or an equivalent orchestration layer handles the reasoning loop.
- The agent is tool-driven and must not fabricate numbers.
- The frontend currently contains static UI states, so the backend must supply data that can replace the hardcoded examples.

- config.py must hold all environment-driven settings.

- stats/analytics_service.py contains analytics aggregation logic used by the analytics controller.
- static/charts/ stores generated chart files returned by query and analytics flows.

If a later implementation needs different internal names, keep this structure as the public contract and map legacy names into it.


## File-by-File Implementation Brief

This is the short, implementation-ready version of the backend. It is written so another AI can generate the full backend without guessing.

### `main.py`
- Create the FastAPI app.
- Register CORS.
- Include all routers under a stable `/api` prefix.
- Initialize logging, DB, and scheduler startup hooks.
- Expose `/api/system/health` and `/api/system/readiness`.

### `config.py`
- Define a single settings object.
- Read `DATABASE_URL`, `JWT_SECRET`, `JWT_ALGORITHM`, `ACCESS_TOKEN_EXPIRE_MINUTES`, `CORS_ORIGINS`, `APP_NAME`, `APP_VERSION`, `CHART_DIR`.
- Provide safe defaults for local development.
- Fail fast when required production settings are missing.

### `controller/auth.py`
- Implement signup, login, and current-user endpoints.
- Use `utils/auth.py` for password hashing and JWT issuance.
- Enforce unique email addresses.
- Return token and user metadata.

### `controller/query.py`
- Accept natural-language questions.
- Load memory and project context.
- Call `engine/orchestrator.py`.
- Persist query logs.
- Return answer, trace, chart URL, and confidence metadata.

### `controller/history.py`
- Return paginated query history.
- Support lookup by history item id.
- Support search and filters.

### `controller/analytics.py`
- Return telemetry, trends, and KPI cards.
- Call `stats/analytics_service.py` for aggregation.

### `controller/notification.py`
- Manage alert rules and notification logs.
- Implement CRUD for alert rules.
- Expose logs for dispatched alerts.
- Keep rule creation separate from rule evaluation.

### `engine/orchestrator.py`
- Decide which tool to use.
- Call SQL, RAG, insight, action, and chart tools.
- Never fabricate values.
- Return structured evidence and final answer text.

### `engine/sql_engine.py`
- Execute read-only SQL only.
- Reject writes and unsafe statements.
- Return rows plus column metadata.

### `engine/insight_engine.py`
- Compute growth, deltas, anomalies, and summaries from structured data.

### `engine/rag_engine.py`
- Retrieve relevant document chunks from vector storage.

### `engine/action_engine.py`
- Create alert rule records from user intent.
- Do not send notifications directly unless called by the scheduler.

### `engine/scheduler.py`
- Run on a timed loop or background worker.
- Fetch active alert rules.
- Evaluate metrics.
- Create `NotificationLog` rows when conditions match.
- Trigger optional delivery.

### `model/db_models.py`
- Define `User`, `Project`, `QueryLog`, `AlertRule`, `NotificationLog`, `SystemMetric`, and `ConversationSession`.
- Keep timestamps and indexes consistent.

### `model/schemas.py`
- Define request and response schemas for auth, query, history, analytics, and notifications.

### `model/llm.py`
- Create and configure the LLM client used by the orchestrator.

### `utils/db.py`
- Create the SQLAlchemy engine and session factory.
- Provide `get_db()` and `init_db()` helpers.

### `utils/auth.py`
- Hash passwords, verify passwords, create tokens, and decode tokens.

### `utils/logger.py`
- Configure structured logging and request correlation IDs.

### `stats/analytics_service.py`
- Aggregate query logs and metrics into dashboard-ready telemetry.

### `static/charts/`
- Store generated chart images that the API can return to the frontend.


## Required Notification Design

The notification system must follow this exact model.

### Core Components
1. `AlertRule` is the rule a user creates.
2. `NotificationLog` stores the outcome of a triggered rule.
3. `notification.py` is the API router for rules and logs.
4. `scheduler.py` is the background evaluator that makes alerts actually work.

### AlertRule table
Use these fields at minimum:
```json
{
  "id": 1,
  "user_id": 101,
  "metric": "monthly_revenue",
  "operator": "<",
  "threshold": 50000,
  "notification_type": "email",
  "quiet_hours_start": "22:00",
  "quiet_hours_end": "07:00",
  "is_active": true,
  "created_at": "..."
}
```

### NotificationLog table
Use these fields at minimum:
```json
{
  "id": 901,
  "rule_id": 1,
  "triggered_value": 42000,
  "status": "delivered",
  "channel": "email",
  "created_at": "..."
}
```

### Notification API endpoints
- `POST /api/notifications/rules` creates a rule.
- `GET /api/notifications/rules` lists rules.
- `PATCH /api/notifications/rules/{id}` updates a rule.
- `DELETE /api/notifications/rules/{id}` disables a rule.
- `GET /api/notifications/logs` returns delivery logs.

### Scheduler flow
1. User creates an alert rule.
2. The rule is stored in the database.
3. The scheduler runs every X minutes.
4. Active rules are fetched.
5. Each rule is evaluated against live SQL or metric data.
6. If the condition is true, create a `NotificationLog` row.
7. Send the notification if delivery is enabled.

### Notification rules
- Quiet hours must be honored.
- Duplicate sends must be prevented.
- Delivery failures must still be logged.
- Rules must remain user-scoped.

## File: server.py

### 1. Purpose
server.py is the runtime entry point when the backend is launched from the command line.
Its job is to create the FastAPI application object, load configuration, register middleware, mount routers, and expose the application instance that Uvicorn imports.
This file is not where business logic belongs.
It is the composition root that wires the backend together.
It should be small, explicit, and deterministic.
It should not perform data analysis, SQL execution, or agent reasoning.
It should only initialize the service graph.
In the current repository, server.py is empty, which means the application cannot boot.
The file therefore needs to become the central startup module.
The app object it exports becomes the top-level integration point for the frontend.
Every API call from React will pass through this file first.
Because of that, this file controls CORS, routers, lifespan hooks, startup checks, and shutdown cleanup.
The file also defines whether the backend is ready, healthy, and reachable by the browser.
If the app is not configured correctly here, none of the other files matter.
The implementation must guarantee that routes are mounted under stable prefixes.
The implementation must also guarantee that error handling is consistent across all endpoints.
A clean server.py makes local development easy and simplifies deployment.
The file should remain focused on orchestration only.
It should not import heavy tool modules until necessary unless startup validation requires it.
The file should support both reload-based development and production deployment.
It should also be the natural place to define a health endpoint if one is not separated into its own router.
The frontend's navbar and sidebar status indicators require this file to expose a live status route.
The landing page and dashboard can use the same route to confirm the system is alive.
Because the frontend currently shows static text, the implementation should still anticipate future real-time status polling.
The file should also ensure the API root can be discovered consistently by the frontend.
If versioning is used, this file should mount routers under /api/v1 or a similar stable prefix.
The route prefix must remain consistent across authentication, queries, history, analytics, and notification endpoints.
This file is also the correct place to attach exception handlers for validation and unexpected runtime errors.
Those handlers keep the frontend from receiving raw Python traces.
The startup object should expose metadata such as title, version, and description.
That metadata helps with OpenAPI docs and internal debugging.
The server entry point is also where startup resource checks can confirm database connectivity and vector store availability.
That allows the system to fail fast rather than hang later during the first user query.
The app should be built to support deterministic startup and graceful shutdown.
That means closing DB sessions, flushing logs, and releasing background task handles.
Because the frontend is browser-based, the app must include CORS configuration that allows the frontend origin during development.
For deployment, the allowed origins should be configurable through environment variables.
The main principle is that server.py orchestrates the backend, but never contains the agent brain itself.

### 2. Responsibilities
- Create the FastAPI application instance.
- Load configuration from config.py.
- Register CORS middleware.
- Register global exception handlers.
- Mount all controllers and route groups.
- Provide health and readiness endpoints.
- Trigger startup checks for database and model dependencies.
- Trigger graceful shutdown cleanup.
- Expose the ASGI app object for Uvicorn.
- Define route prefixes and API versioning policy.
- Preserve a clean separation between wiring and business logic.
- Ensure the frontend can always reach the backend through a predictable base URL.
- Support OpenAPI generation for internal testing and debugging.
- Initialize shared dependencies such as logger configuration.
- Optionally initialize background workers or scheduled jobs.
- Avoid hardcoding secrets, paths, or deployment-specific values.

### 3. Internal Components
- FastAPI app instance with title, description, version, docs URL, and openapi URL.
- CORS middleware that reads allowed origins from config.
- Router inclusion statements for all controller modules.
- Lifespan context manager or startup/shutdown event handlers.
- Exception handlers for HTTPException, validation errors, and unhandled exceptions.
- Health and readiness response models if the route is declared here.
- Optional root redirect or API info endpoint.
- Optional static file mounting for generated charts.
- Optional background task registration hooks.
- Logging bootstrap logic that ensures request logs are structured and correlated.
- Database warmup logic that checks connectivity without mutating data.
- Model or embedding warmup logic if a local model is used.

### 4. Data Flow
- The process starts when Uvicorn imports app from server.py.
- server.py loads settings and initializes logging.
- It creates the FastAPI app with metadata.
- It attaches CORS middleware so browser requests are allowed.
- It mounts routers from controller modules.
- On startup, it optionally verifies that the database and vector index are reachable.
- A browser request from the frontend hits the app through one of the registered routes.
- The request then flows into the appropriate controller and deeper backend layers.
- On shutdown, the app closes open database engines, flushes logs, and releases any held resources.
- If a health endpoint is queried, server.py returns readiness metadata and dependency status.

### 5. Function-Level Breakdown
- create_app(): constructs and returns the FastAPI application instance.
- register_middleware(app): attaches CORS and any request tracing middleware.
- register_routers(app): imports and includes controller routers.
- register_exception_handlers(app): centralizes error formatting.
- startup_event(): performs dependency checks and warmups.
- shutdown_event(): closes engines and flushes output buffers.
- health_route(): returns service health information.
- readiness_route(): returns whether the backend can serve requests.
- mount_static_assets(app): exposes generated charts or documents if needed.
- init_logging(): prepares the global logger before request handling begins.
- validate_environment(): confirms required environment variables are present.

### 6. Integration with Other Files
- Uses config.py for environment and deployment settings.
- Uses utils/logger.py for request and system logging.
- Uses utils/db.py for database engine and session initialization.
- Includes controller/auth_router.py for login and signup routes.
- Includes controller/project_router.py for project listing and workspace metadata.
- Includes controller/query_router.py for the natural-language query endpoint.
- Includes controller/history_router.py for audit trail retrieval.
- Includes controller/analytics_router.py for telemetry and charts.
- Includes controller/notification_router.py for alerts and dispatch logs.
- May include routes/__init__.py if routers are aggregated in a registry pattern.
- Depends on model/schemas.py so responses are serialized consistently.
- Depends on model/db_models.py if startup needs to create tables.
- May depend on agent/orchestrator.py for startup validation of tool availability.

### 8. Edge Cases & Failure Handling
- Missing environment variables should fail fast at startup.
- Database unavailable at startup should return readiness=false and log an actionable error.
- A bad router import should stop app boot rather than produce partially mounted routes.
- Validation errors should return a stable JSON shape the frontend can read.
- Unexpected exceptions should be converted into sanitized 500 responses.
- Duplicate route prefixes should be detected during integration testing.
- Static asset directory missing should not crash the app if static files are optional.
- CORS misconfiguration should be visible in startup logs.
- If a dependency warmup times out, the app should still start in degraded mode only if explicitly allowed.
- Unhandled startup failures should be evident in Uvicorn output and structured logs.

### 9. Security Considerations
- Never embed secrets directly in the source file.
- Restrict CORS origins to trusted frontend hosts.
- Avoid exposing internal stack traces to the browser.
- Keep route registration explicit to prevent accidental exposure of debug endpoints.
- Ensure health endpoints do not leak sensitive configuration values.
- Sanitize startup error messages before returning them to clients.
- If static files are mounted, ensure only safe directories are exposed.
- Do not use server.py to store tokens, passwords, or database credentials.
- Protect admin-only diagnostics behind authentication if they are ever added.
- Make logs useful but not credential-bearing.

### 10. Scalability Notes
- Keeping server.py thin makes scaling easier because workers can be added without code duplication.
- Route mounting should remain modular so new APIs can be added without touching the composition root heavily.
- Lifespan warmups should be lightweight to avoid slow container startup.
- Health checks should be cheap enough for load balancers and uptime probes.
- CORS and middleware layers should remain minimal to reduce per-request overhead.
- Static file serving should be disabled or offloaded when using a CDN or object storage.
- The application factory pattern makes testing and multi-environment deployment easier.
- If the backend later becomes multi-process, startup code must remain idempotent.
- Shared state should not be stored in module globals unless read-only.


## File: utils/db.py

### 1. Purpose
utils/db.py owns database connectivity, session creation, and transaction lifecycle management.
It is the low-level bridge between application code and persistent storage.
All database-backed routers and services should go through this module rather than creating ad hoc connections.
The file is currently empty, which means no database access is available yet.
This file must become the single source of truth for engine creation, session factories, and dependency injection.
It should support local SQLite for development and a more scalable database for production.
It should hide connection details from controllers and services.
It should also define the dependency function FastAPI uses to inject DB sessions into routes.
Because the frontend relies on history, analytics, notifications, users, and project metadata, this module is foundational.
It also supports logging query execution and storing alert rules.
If the database layer is wrong, the entire agent history and audit trail become unreliable.
The module must therefore be robust, explicit, and easy to test.
It should own the engine, sessionmaker, and base metadata utilities.
It should not contain ORM models themselves.
It should not mix schema definitions with connection setup.
The file should expose a create_engine configuration that can be swapped via environment variables.
It should also make table creation possible for local bootstrapping.
If using SQLite, it must ensure the required connect arguments are set correctly.
If using PostgreSQL later, it must support pooling and sane connection settings.
This module should be the place where migrations are prepared, even if Alembic is added later.
The get_db dependency should yield a session and guarantee close on completion.
The module should preserve one session per request semantics.
The code should be compatible with synchronous SQLAlchemy usage because many endpoints will use synchronous ORM access.
If async database access is adopted later, the module should isolate the migration path.
The interface should stay simple enough that routers do not need to know the engine flavor.
The frontend never directly sees this file, but every persisted feature depends on it.
That includes auth data, query logs, history rows, alert rules, and aggregated dashboard metrics.
The module should also support a read-only connection mode for analytical endpoints if needed.
It should help prevent session leakage under error conditions.
It should also be the place where database file paths are normalized on Windows and Unix.
The implementation must explicitly handle SQLite threading behavior.
The module can also expose helper functions for initializing sample data in development.
Those helpers should never run in production automatically.
The goal is to make persistence predictable and safe.

### 2. Responsibilities
- Create the SQLAlchemy engine.
- Create the session factory.
- Provide a get_db dependency for FastAPI.
- Provide optional initialization helpers for tables and seed data.

- Close sessions reliably after each request.
- Support future migration to PostgreSQL.
- Optionally expose the declarative base class.
- Optionally expose transaction helpers.
- Avoid importing routers or agent code.
- Keep the persistence layer isolated from presentation logic.

### 3. Internal Components

- SessionLocal factory configured with autocommit=False and autoflush=False.
- Base declarative class.
- get_db() generator dependency.
- init_db() helper to create all tables.
- maybe_drop_and_recreate() helper for test environments only.
- connection argument normalization for SQLite.
- optional ping/health helper for startup checks.
- optional transaction context manager.

### 4. Data Flow

- The URL is normalized depending on whether SQLite or another database is used.
- The engine is instantiated.

### 8. Edge Cases & Failure Handling

## End-to-End Execution Flow

1. The user opens the Query page in the frontend and enters a natural-language question.
2. The frontend sends POST /api/query with the query text, optional project_id, selected sources, and session_id.
3. controller/query_router.py validates the payload and authenticates the user.
4. The router fetches recent conversation memory from agent/memory.py.
5. The router loads project context if the query is scoped to a workspace.
6. The router passes the request, memory, and scope hints to agent/orchestrator.py.
7. The orchestrator builds the system prompt using agent/prompts.py.
8. The orchestrator asks the model to decide the next action.
9. If the model selects SQL, the orchestrator invokes tools/sql_engine.py.
10. tools/sql_engine.py validates the query as read-only and executes it.
11. The returned rows are fed into tools/insight_engine.py if the agent requests statistical analysis.
12. If document context is needed, tools/rag_engine.py retrieves relevant chunks from the vector index.
13. If a chart would help, tools/viz_engine.py renders an image and returns a static URL.
14. If the user requested an alert or automation, tools/action_engine.py creates the alert rule.
15. The orchestrator collects every observation and assembles a final answer.
16. The orchestrator returns the answer text, execution trace, and any artifacts to controller/query_router.py.
17. The query router saves a QueryLog record through utils/db.py and model/db_models.py.
18. The router returns the final JSON payload to the frontend.
19. The frontend displays the answer, chart, and trace, and the same record becomes available in the History page.
20. If the query produced a persistent alert, the Notification Center can later show the associated rule and dispatch log.
21. If the request changed system metrics, analytics routes can aggregate the new operational data.
22. The same backend state also feeds the dashboard cards and system health indicators.

Notification-only execution flow:
1. User creates rule.
2. Rule is stored in AlertRule.
3. Scheduler runs every X minutes.
4. Evaluator fetches active rules.
5. Evaluator checks data via SQL or metrics tables.
6. Evaluator evaluates operator and threshold.
7. If true, evaluator creates NotificationLog.
8. Notification is sent by configured channel when dispatch is enabled.

Implementation priority order:
- Build config.py, utils/db.py, and model/db_models.py first.
- Add utils/auth.py and controller/auth_router.py next.
- Add controller/project_router.py and controller/query_router.py after persistence exists.
- Add agent/orchestrator.py and the tool modules once the API contracts are defined.
- Add history, analytics, and notification routers after query logs and alert rules are writable.
- Add jobs/notification_evaluator.py immediately after notification router CRUD is working.
- Wire server.py last so the complete backend surface is mounted cleanly.

Operational rule for this backend:
- If the backend cannot explain where a value came from, it should not present that value as fact.
- If a number is displayed, it should either come from SQL, a metric table, or a computation trace.
- If a chart is shown, the response should identify the source data and render path.
- If a rule is created, the record must be persisted and auditable.
- If a history entry exists, it must correspond to an actual request lifecycle.
- If a health badge is green, the health endpoint must be able to justify it.
